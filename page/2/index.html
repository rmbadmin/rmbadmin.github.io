【小白向】Unity引擎DLL文件的引用方式

发表于 2017-05-04 更新于 2019-09-21 分类于 Csharp ， 其它语言

最近需要在Unity引擎中使用网络通信，奈何直接引用.NET的TCP Client类似乎没什么效果，所以想到打包成DLL再调用，于是在网上找了如何在Unity中使用DLL文件，但是似乎都没什么用。刚刚在Unity的官方文档里找到了“官方”的解决方案，整理如下。 DLL**文件的生成** 关于编译器： 如果你要编译的模块不依赖与Unity的API，则选择合适的编译器编译即可，如果需要与Unity引擎交互，可以引用UnityEngine.dll和UnityEditor.dll这两个DLL文件。它们的位置为：

Windows：Program Files\Unity\Editor\Data\Managed\

macOS：Applications/Unity/Unity.app/Contents/Frameworks/Managed/

注：根据我遇到的情况，编译的DLL如果使用了.NET库，.NET版本不要高于3.5。 DLL**文件的使用** 将DLL文件拷贝至Unity引擎的Assets文件夹中，然后在C#脚本的开头处加入using。如，对与DLLTest.dll来说，应加入“using DLLTest;”。当将脚本添加到场景中的一个对象上后，点击播放即可发现脚本已经在工作了。 注：网上有很多说要放在什么Scripts文件夹或者Plugins文件夹下的，但是根据我的情况，放在那些地方都会导致编译器找不到dll，还是要放在根目录下。就是这个地方把我坑惨了…… 关于使用MonoDevelop调试，需要将mdb文件复制进Assets或Plugins文件夹。VS编译的DLL没有mdb文件，可以使用工具进行转换，位置如下：

Program Files\Unity\Editor\Data\Mono\lib\mono\2.0\pdb2mdb.exe

其命令行参数为pdb文件，会在同一目录下生成对应的mdb。 打开Tools菜单（Windows）或MonoDevelop-Unity menu (macOS)，调整设置Add-in Manager > Installed tab > Unity > select Mono Soft Debugger Support for Unity > Enable之后即可开始调试。

【小白向】UWP开发中的Json处理

发表于 2017-03-23 更新于 2019-09-21 分类于 Csharp ， 编程

首先声明，本人纯粹小白，之前没正经学过C#，没正经学过UWP，也没接触过Json，本文涉及的内容中可能充满了错误，所以请读者在阅读时自行判断。 事情的起因是这样的，我想写一个第三方的客户端，然后发现从服务器读回来的数据全都是Json格式的，我要想简单的提取出所需的信息就需要处理Json内容（否则我就只能一点点处理字符串了，那真的会非常麻烦）。 本来网上有关C#序列化或反序列化Json的博文有很多，但是不知道是我没搜索合适的关键字还是怎样，并没有找到合适的内容，很多都是针对传统桌面程序的库，后来忘了在哪（可能是MSDN吧）突然发现了一个叫Windows.Data.Json的命名空间，一下子点醒了我，而且这个也是非常好用，非常傻瓜式的。 其中我用到的几个类大概是JsonObject和JsonArray，几个方法大概是Parse、GetObjectAt和GetNamedString等一大票GetNamed开头的方法，每个的使用方法大同小异，大概一说，相信各位编程老手在VS里一敲，靠着Intellisense就足以搞清其用法了。 JsonObject是最常用的一个类了，其对象就是一个Json结构，可以通过Parse方法，传入一个字符串（也就是我从服务器上接收到的那堆东西）来初始化 当我要从中得到某个字段的值时，根据字段的属性，是字符串还是个JSon数组或是别的什么，分别调用GetNamedString、GetNamedArray方法即可，参数是一个字符串，存放着字段的名字。 对于Json数组，也就是用“[”和“]”包裹起来的那堆东西，直接使用JsonArray类型就可以处理，然后分别提取其中的某个对象可以使用GetXXXXAt这个方法，获取到对象之后的操作，就和之前的没什么区别了。

Xna与WinForm同时工作的实现方法

发表于 2017-01-19 更新于 2019-09-21 分类于 Csharp ， 心得技巧 ， 编程

最近我游戏项目的引擎设计告一段落，是时候开始设计具体的游戏内容。而其首先要解决的就是地图设计问题。 在有一个比较好的地图生成算法之前，地图还是得手工绘制，因此想要制作一个地图编辑器来方便进行地图设计。在经过一番比对后，我选择Xna+WinForm的方式来设计编辑器，然后又经过一番寻找，最终找到了官方给的一个例子。链接将在最后放上。 OK，那么先来解读官方给的例子，再设计我们自己的项目 官方的例子是将Xna以控件的形式加入到WinForm中去，这刚好符合我的想法 注：由于本人技术原因，一些东西的工作原理并不是很清楚，所以基本上是依葫芦画瓢←_← 1.解读部分 首先看到GraphicsDeviceService类。 这个类实现了IGraphicsDeviceService接口，根据注释，这个接口提供了对GraphicsDevice对象的索引。这个接口包含1个属性和四个事件，他们分别是：

GraphicsDevice

GraphicsDevice { get; }
event EventHandler DeviceCreated;
event EventHandler DeviceDisposing;
event EventHandler DeviceReset;
event EventHandler DeviceResetting;

第一个属性就是要提供的GraphicsDevice对象，后面四个事件的作用从名字就可以知道 在GraphicsDeviceService类的实现中，GraphicsDevice属性返回一个GraphicsDevice类型的字段graphicsDevice，这个字段的初始化实现在类的私有构造函数中。其余四个事件仅仅是简单的实现了一下。 通过公有静态方法AddRef以及私有静态字段singletonInstance来看，此处实现了一个单例模式。虽然不知道具体原因，但我猜测是由于GraphicsDevice不能有多个实例，因此采用单例模式来避免出现问题。 现在将目光转向私有的构造函数：

GraphicsDeviceService(IntPtr windowHandle, int width, int height)
{
parameters = new PresentationParameters();

parameters.BackBufferWidth = Math.Max(width, 1);
parameters.BackBufferHeight = Math.Max(height, 1);
parameters.BackBufferFormat = SurfaceFormat.Color;
parameters.DepthStencilFormat = DepthFormat.Depth24;
parameters.DeviceWindowHandle = windowHandle;
parameters.PresentationInterval = PresentInterval.Immediate;
parameters.IsFullScreen = false;

graphicsDevice = new GraphicsDevice(GraphicsAdapter.DefaultAdapter,
GraphicsProfile.Reach,
parameters);
}

构造函数中初始化了graphicsDevice字段，根据其构造函数签名，初始化了字段parameters，并且设置了此字段的各项属性。包括： 后背缓冲区的尺寸（BackBufferWidth，BackBufferHeight属性） 后背缓冲区的格式（BackBufferFormat属性，值为SurfaceFormat.Color，解释为：32位的ARGB色彩格式） 深度模板数据格式（DepthStencilFormat属性，值为DepthFormat.Depth24，解释为：包括24-bit的深度数据。看起来是用于计算图像深度的） 设备窗口句柄（DeviceWindowHandle属性，值为构造函数的参数windowHandle，也就是之后我们设计的控件的句柄） 呈现的时间间隔（PresentationInterval属性，值为PresentInterval.Immediate，不是很清楚这是干什么的） 是否全屏（IsFullScreen属性，值为false） 然后是另外两个参数，分别是GraphicsAdapter类型和GraphicsProfile类型，值分别为GraphicsAdapter.DefaultAdapter和GraphicsProfile.Reach。关于第二个参数，应该就是Xna工程中在Properties里设置的Game profile。 我们关注的东西基本就是以上那些，其余的几个方法就简单介绍一下：

public static GraphicsDeviceService AddRef(IntPtr windowHandle,int width, int height)

返回对singletonInstance的引用，同时使引用计数+1。如果之前计数为0，就用传进来的三个参数初始化singletonInstance。

public void Release(bool disposing)

使引用计数-1，计数为0时根据disposing的值选择是否销毁graphicsDevice

public void ResetDevice(int width, int height)

重新设置parameters字段的后背缓冲区尺寸，并重置graphicsDevice字段。 接下来，看到我们的重点：GraphicsDeviceControl类 此类继承至Control类，同时又是抽象类，是之后SpriteFontControl类和SpinningTriangleControl类的父类。 它包含了个字段和属性：

GraphicsDeviceService graphicsDeviceService;

ServiceContainer services = new ServiceContainer();

public GraphicsDevice GraphicsDevice
{
get { return graphicsDeviceService.GraphicsDevice; }
}

public ServiceContainer Services
{
get { return services; }
}

第一个字段和第一个属性不需要解释了，第二个字段则是一个实现了IServiceProvider接口的类的实例。IServiceProvider是FCL（**F**ramework **C**lass **L**ibrary，即Framework类库）内定义的接口，按照官方的说明，是

定义一种检索服务对象的机制，服务对象是为其他对象提供自定义支持的对象。

看起来是一种解耦的设计，不过不是很清楚在哪些地方发挥了作用。在这个例子中，它将作为ContentManager构造函数的一个参数来创建ContentManager类型实例。 GraphicsDeviceControl类重写了几个Control父类的方法，其中值得注意的是OnCreateControl、OnPaint、OnPaintBackground这三个方法。

protected override void OnCreateControl()

在OnCreateControl方法中，代码用父类的Handle、ClientSize.Width、ClientSize.Height这三个属性初始化了graphicsDeviceService字段，并且将其作为参数调用了services.AddService方法，最后调用了定义的抽象方法Initialize。

protected override void OnPaint(PaintEventArgs e)

在OnPaint方法中，代码首先判断了调用定义的BeginDraw方法的返回值是否为空，然后依次调用Draw、EndDraw方法。如果不为空，则调用PaintUsingSystemDrawing方法用GDI+打印出错误信息。

protected override void OnPaintBackground(PaintEventArgs pevent)

OnPaintBackground方法比较特殊，它什么内容也没有，目的是防止绘制的时候闪屏。

string BeginDraw(）
void EndDraw()
string HandleDeviceReset()

在BeginDraw方法中，调用了定义的HandleDeviceReset方法，这个方法用于处理graphicsDevice的重置问题，根据其GraphicsDeviceStatus属性的值，分别选择返回错误信息或者重新设置后背缓冲区的尺寸。紧接着，BeginDraw方法又创建了Viewport类型的实例，这个类主要管理显示的区域。代码设置了要显示的区域在缓冲区中的坐标以及大小，并且设置了层深值的取值范围。最后将其传给了graphicsDevice的Viewport属性。之后再调用定义的抽象方法Draw，最后调用EndDraw方法将绘制结果呈现出来。 至此，所有的准备工作基本完成，接下来就是创建GraphicsDeviceControl类的子类SpriteFontControl和SpinningTriangleControl。这两个类中的代码都是很明显的，我就不再多说。 解读到此为止。 2.实现自己的项目 这个没什么好多讲的，基本抄袭官方代码按照解读来做就好了←_← 才不会告诉你是因为懒所以不想写了←_← 3.最后的效果： 看起来好像和一般的Xna工程一样嘛 官方例子的链接： http://pan.baidu.com/s/1gf7MHMF

毕竟我是小白嘛，和站长大大没法比，所以只好写一些这类的粗浅文章了，如果有帮助到初学的朋友就好了。 关于static关键字的用法，网上写的文章大多冗长，看了以后感觉所有的情况都可以用最简单的几句话来推导，所以总结如下。

1.static限定的变量在全局数据段中，仅限本文件访问，而且只会初始化一次（缺省为0），整个程序只有一份。

2.static修饰的函数，只在声明它的文件中可见。对于对象的成员函数如果被它修饰，那么它针对的是整个类，不再具有缺省的this指针。

只是小白的总结，如果有错误还请指正…… 求轻喷……
